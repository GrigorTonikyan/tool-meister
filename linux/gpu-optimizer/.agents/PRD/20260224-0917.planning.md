## 2. Tech Stack

| Layer | Technology | Rationale |
|---|---|---|
| Runtime | **Bun** | Project mandate; sole runtime/package manager |
| Language | **TypeScript** (strict mode) | Existing; type safety across modules |
| TUI Framework | **`terminal-kit`** | See §2.1 below |
| TUI Types | **`@types/terminal-kit`** | TypeScript definitions for terminal-kit |
| Existing deps (retained) | `@clack/prompts`, `picocolors`, `zod` | `@clack/prompts` used for CLI-passthrough mode; `picocolors` for diff coloring; `zod` for config validation |
| Config Format | **JSON** with `zod` schema validation | Simple, native `JSON.parse`/`JSON.stringify`, validated on load |

### 2.1 TUI Framework Decision: `terminal-kit`

> [!IMPORTANT]
> **Why `terminal-kit` over alternatives:**
>
> | Option | Verdict |
> |---|---|
> | `ink` | ❌ React-based — explicitly rejected by user |
> | `blessed` / `neo-blessed` | ❌ Unmaintained (10+ years stale) |
> | `@unblessed/node` | ❌ Alpha stage, not production-ready |
> | `@mariozechner/pi-tui` | ❌ Niche, differential-rendering focus, small ecosystem |
> | `Bun.Terminal` API | ❌ PTY-level (spawn subprocesses into TTY), not a widget framework |
> | **`terminal-kit`** | ✅ Mature, event-driven, full widget system, keyboard + mouse, screen buffers, Bun-compatible (confirmed fixed Jan 2025) |
>
> `terminal-kit` provides: single-column menus, grid menus, input fields, progress bars, tables, screen/off-screen buffers, and a Document Model for building persistent layouts — all event-driven without a React-like reconciler.

## 3. Core Architecture: Decoupled Layers

The monolithic `src/index.ts` must be decomposed into three distinct layers:

```
┌─────────────────────────────────────────────┐
│              Entry Point (src/index.ts)      │
│   Parses CLI args → routes to TUI or CLI    │
└──────────────┬──────────────────┬────────────┘
               │                  │
     ┌─────────▼──────┐  ┌───────▼──────────┐
     │   TUI Layer    │  │   CLI Layer      │
     │  src/tui/      │  │  src/cli/        │
     │  terminal-kit  │  │  @clack/prompts  │
     └───────┬────────┘  └───────┬──────────┘
             │                    │
     ┌───────▼────────────────────▼──────────┐
     │          Controller Layer              │
     │         src/controllers/               │
     │  Orchestrates engine calls, exposes    │
     │  clean async API for both UI layers    │
     └────────────────┬──────────────────────┘
                      │
     ┌────────────────▼──────────────────────┐
     │           Engine Layer                 │
     │  src/discovery/ | src/engine/          │
     │  src/utils/     | src/types.ts         │
     │  (UNCHANGED — zero modifications)      │
     └────────────────────────────────────────┘
```

> [!NOTE]
> The existing engine layer (`src/discovery/`, `src/engine/`, `src/utils/`, `src/types.ts`) remains **untouched** except for type extensions in `src/types.ts`. No engine logic changes.

## 4. XDG-Compliant File Paths

All user data must follow the [XDG Base Directory Specification](https://specifications.freedesktop.org/basedir-spec/latest/):

| Purpose | XDG Variable | Default Path | App Subdirectory |
|---|---|---|---|
| Configuration (settings) | `XDG_CONFIG_HOME` | `~/.config` | `gpu-optimizer/config.json` |
| State data (backups) | `XDG_STATE_HOME` | `~/.local/state` | `gpu-optimizer/backups/` |
| Log files | `XDG_STATE_HOME` | `~/.local/state` | `gpu-optimizer/logs/` |

> [!IMPORTANT]
> The existing backup engine uses `~/.local/state/gpu-optimizer/backups/` which **already** aligns with `XDG_STATE_HOME`. No change needed there. Only the new config and log paths need implementation using the XDG resolution pattern.

### 4.1 XDG Resolution Logic

```typescript
function resolveXdgPath(envVar: string, fallback: string, ...segments: string[]): string {
  const base = process.env[envVar] || join(homedir(), fallback);
  return join(base, 'gpu-optimizer', ...segments);
}
```

## 6. Telemetry Strategy: Point-in-Time Snapshots

> [!NOTE]
> **No live polling / shadow-graph diffs.** Telemetry data (CPU usage, temperatures, GPU stats, memory) is collected as a **point-in-time snapshot** each time the user navigates to a status screen. A manual "Refresh" action (keybind `r`) re-collects the snapshot.

### 6.1 Data Sources

| Metric | Source | Method |
|---|---|---|
| CPU model | `/proc/cpuinfo` | File read, parse `model name` |
| CPU cores | `/proc/cpuinfo` or `nproc` | File read or shell |
| CPU usage | `/proc/stat` | Two reads ~100ms apart, compute delta |
| CPU temperature | `/sys/class/hwmon/*/temp*_input` | File read, match `coretemp`/`k10temp` |
| GPU temperature | `/sys/class/drm/card*/device/hwmon/*/temp*_input` | File read |
| GPU utilization | `nvidia-smi` (NVIDIA) or `/sys/class/drm/card*/device/gpu_busy_percent` (AMD) | Shell or file read |
| VRAM | `nvidia-smi` (NVIDIA) or `/sys/class/drm/card*/device/mem_info_vram_*` (AMD) | Shell or file read |
| Memory stats | `/proc/meminfo` | File read, parse `MemTotal`/`MemFree`/`MemAvailable` |

* [ ] **11.1** Create `src/cli/index.ts` — Argument parser using `process.argv` (no external dep). Supports:

  - `--help` / `-h` — Print usage
